# 编码转换测试场景

本文档概述了自动化编码转换工具将要测试的主要场景。重点关注常见的实际应用场景，主要涉及与 UTF-8、UTF-16 以及主要的区域性传统编码之间的相互转换。

## 核心场景

针对源文件的通用测试流程如下：

1.  **检测源编码：** 识别 `tests/from` 目录中文件的原始编码。
2.  **正向转换：** 将源文件转换为一个主要目标编码（通常是 UTF-8，除非源文件已经是 UTF-8，这种情况下可能会使用像 GBK 这样的常见传统编码）。保存到 `tests/to` 目录。
3.  **反向转换：** 将 `tests/to` 目录中的文件从主要目标编码转换回其*最初检测到的源编码*。保存到 `tests/back` 目录。
4.  **比较：** 比较原始源文件 (`tests/from`) 与反向转换后的文件 (`tests/back`)，检查字节级别的一致性。

## 场景矩阵

| 场景 ID     | 源编码类型                | 主要目标编码 (正向) | 反向目标编码     | 测试关键点                                      | 预期可逆性     |
| :---------- | :-------------------------- | :---------------------- | :----------------- | :------------------------------------------------ | :------------- |
| SCENARIO-01 | 常见传统编码 (GBK, BIG5)  | UTF-8                   | 原始传统编码     | 传统编码 -> UTF-8 -> 传统编码 往返转换            | 高             |
| SCENARIO-02 | 常见传统编码 (Shift-JIS)  | UTF-8                   | 原始传统编码     | 日语传统编码 -> UTF-8 -> 传统编码               | 高             |
| SCENARIO-03 | 常见传统编码 (EUC-KR)     | UTF-8                   | 原始传统编码     | 韩语传统编码 -> UTF-8 -> 传统编码               | 高             |
| SCENARIO-04 | 常见传统编码 (ISO-8859-x) | UTF-8                   | 原始传统编码     | 欧洲传统编码 -> UTF-8 -> 传统编码               | 高             |
| SCENARIO-05 | 常见传统编码 (Windows-*)  | UTF-8                   | 原始传统编码     | Windows 传统编码 -> UTF-8 -> 传统编码           | 高             |
| SCENARIO-06 | UTF-8 (有/无 BOM)         | GBK                     | UTF-8              | UTF-8 -> 特定传统编码 -> UTF-8 往返转换         | 中/高 (取决于内容) |
| SCENARIO-07 | UTF-16LE (有/无 BOM)      | UTF-8                   | UTF-16LE           | UTF-16LE -> UTF-8 -> UTF-16LE                 | 高             |
| SCENARIO-08 | UTF-16BE (有/无 BOM)      | UTF-8                   | UTF-16BE           | UTF-16BE -> UTF-8 -> UTF-16BE                 | 高             |
| SCENARIO-09 | ASCII                     | UTF-8                   | ASCII              | 基本 7 位 ASCII 一致性                          | 高             |
| SCENARIO-10 | 内容混合的文件            | UTF-8                   | 原始编码(尽力)   | 处理可能并非单一编码的文件                      | 低 / 跳过      |
| SCENARIO-11 | 不可逆的传统编码          | UTF-8                   | 原始传统编码     | 识别已知的不可逆转换 (例如 HKSCS)               | 低 / 失败      |

**注意:**

*   文件名中明确包含 "IRREVERSIBLE" 的文件，在通过某些传统编码进行往返转换后，预期会在比较步骤失败。
*   正向转换的主要目标通常是 UTF-8 (无 BOM)，因为它是最常见的中间格式。如果源已经是 UTF-8，则可能使用 GBK 作为示例目标传统编码。
*   反向转换的目标将是*检测到的*源编码。如果检测失败，反向转换步骤可能会被跳过。
*   BOM 处理：转换*到* UTF-8 时，除非源是 UTF-16，通常省略 BOM。当从传统编码转换*回* UTF-8 时，为了清晰起见可能会添加 BOM，但理想情况下，如果核心内容相同，比较应能处理 BOM 的存在与否。转换回 UTF-16 时，通常应添加 BOM。
